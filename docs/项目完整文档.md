## 1. 项目概述

**codex-claude-orchestrator** 是一个本地化的多 Agent 编排系统，用于管理和协调多个 Claude Code 工作实例。系统以 tmux 会话为运行容器，通过监督者模式（Supervisor Pattern）实现对抗性验证、挑战-修复循环和验收门控。

### 核心定位

- **Codex-first 架构**: 以 Codex 编排器为核心，管理多个 Claude Code worker 实例
- **本地运行**: 所有组件在本地机器上运行，不依赖云服务
- **契约驱动**: Worker 通过 `WorkerContract` 定义能力、权限、工作空间策略和验收标准
- **事件溯源 (V4)**: 最新版本采用事件溯源架构，所有状态变更为不可变事件

### 入口点

```
orchestrator = "codex_claude_orchestrator.cli:main"
```

定义在 `pyproject.toml` 中，通过 `orchestrator` 命令行工具调用。

## 2. 核心功能

### 2.1 多 Agent 编排

系统可以同时管理多个 Claude Code 工作实例，每个实例运行在独立的 tmux 窗格中。支持三种预设角色组合：

| 模式 | 角色组成 | 适用场景 |
|------|---------|---------|
| quick | 仅 implementer | 简单修改 |
| standard | explorer + implementer | 常规开发任务 |
| full | explorer + implementer + reviewer | 复杂任务，需要代码审查 |

### 2.2 动态决策引擎

`CrewDecisionPolicy` 根据当前快照动态决定下一步操作，支持动态生成以下 Worker 类型：

- **context_scout**: 上下文侦察，分析代码库
- **source_write**: 源码编写/修改
- **patch_auditor**: 补丁审计
- **failure_analyst**: 失败分析
- **guardrail_maintainer**: 护栏维护
- **browser_flow**: 浏览器端到端测试

> **规划中**: 决策引擎将升级为 LLM Supervisor 模式。Codex (Claude Code) 作为 supervisor，通过 MCP tools 管理 Worker，规则引擎保留为 fallback。详见 [LLM Supervisor MCP 设计](specs/2026-05-05-llm-supervisor-mcp-design.md)。

### 2.3 对抗性验证循环

系统实现了一个完整的挑战-修复-验证循环：

1. Worker 完成任务后，监督者运行验证命令
2. 验证失败时，系统自动发出挑战（challenge）并要求修复（repair）
3. Worker 尝试修复后再次验证
4. 三次失败后产生学习笔记（learning note）和护栏候选（guardrail candidate）

### 2.4 安全策略门控

- **命令策略**: 阻止 `rm -rf`、`git reset --hard`、`git clean -fd` 等破坏性命令
- **Shell 内联执行**: 阻止 `python -c`、`node -e`、`sh -c` 等内联执行
- **路径保护**: 保护 `.env`、`.git/`、`secrets/`、`*.pem`、`*.key` 等敏感路径
- **写入范围门控**: Worker 只能修改其分配的写入范围内的文件

### 2.5 Git Worktree 隔离

Implementer 类型的 Worker 获得独立的 Git worktree，分支命名规则为 `codex/{crew_id}-{worker_id}`，确保多个 Worker 的代码修改互不干扰。

### 2.6 黑板模式（Blackboard Pattern）

所有 Worker 通过共享黑板交换信息，条目类型包括：`fact`（事实）、`claim`（声明）、`question`（问题）、`risk`（风险）、`patch`（补丁）、`verification`（验证结果）、`review`（审查意见）、`decision`（决策）。

> **当前状态**: 黑板作为审计日志使用，写入后未被自动消费。规划中通过 MCP Server 的 Context Layer 进行过滤和摘要，使其成为 Codex supervisor 的有效工作记忆。

### 2.7 治理式学习系统

- 自动从重复失败中创建学习笔记和护栏候选
- 跟踪 Worker 质量评分
- 新技能和护栏需要人工审批后才能激活
- 安全扫描：检测 API 密钥、私钥、策略绕过模式

### 2.8 Worker 间消息传递

通过 `AgentMessageBus` 实现 Worker 间通信，消息类型：`handoff`（交接）、`question`（提问）、`answer`（回答）、`evidence`（证据）、`status`（状态）。消息通过 `<<<CODEX_MESSAGE ...>>>` 标记块从 Worker 输出中解析。

### 2.9 可视化控制台

内置 HTTP 服务器提供 Web UI，端点包括：
- `/` — 主页面
- `/api/state` — 整体状态
- `/api/sessions/{id}` — 会话详情
- `/api/runs/{id}` — 运行详情
- `/api/skills/{id}` — 技能详情

### 4.2 两代运行时架构

#### V3 — 基于 JSONL 文件的状态持久化

- 状态存储在 JSONL 文件中（append-only 流）：`workers.jsonl`、`events.jsonl`、`decisions.jsonl`、`messages.jsonl` 等
- 可变状态存储在 JSON 文件中：`crew.json`、`tasks.json`、`team_snapshot.json`
- 通过 `CrewRecorder` 类管理，支持乐观锁的状态转换

#### V4 — 事件溯源架构

- 所有状态变更为不可变事件，存储在事件存储中（SQLite 或 PostgreSQL）
- 通过投影（Projection）从事件流重建物化视图
- 支持幂等追加（idempotent append），使用内容摘要作为幂等键
- 事件类型包括：`crew.started`、`worker.registered`、`turn.delivered`、`turn.completed`、`scope.evaluated`、`review.verdict`、`challenge.issued` 等

### 4.3 关键设计模式

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| 多 Agent 编排 | CrewController | 管理多个 Claude Code Worker 实例 |
| LLM Supervisor | MCP Server (规划中) | Codex 作为 supervisor，通过 MCP tools 管理 Worker |
| 战略/战术分层 | Supervision Loop | 战略决策由 LLM，战术执行由自动循环 |
| Context Layer | MCP Server (规划中) | 摘要/过滤/压缩，防止上下文膨胀 |
| 事件溯源 | V4 EventStore | 不可变事件流 + 投影重建状态 |
| 契约驱动 | WorkerContract | Worker 通过契约定义能力与权限 |
| 黑板模式 | BlackboardEntry | 共享知识库，通过 Context Layer 消费 |
| 策略门控 | PolicyGate / WriteScopeGate | 安全策略强制执行 |
| 对抗性验证 | AdversarialEvaluator | 挑战-修复-验证循环 |
| 乐观锁 | transition_worker_status | CAS 语义的 Worker 状态转换 |
| 工厂模式 | build_v4_event_store | 根据环境选择后端 |
| 投影模式 | CrewProjection.from_events | 从事件流重建物化视图 |
| Protocol 接口 | EventStore / RuntimeAdapter | Python Protocol 定义契约 |

### 5.2 核心层 (`core/`)

#### `models.py` — 核心领域模型

定义 V1/V2 运行时的领域模型：

- `TaskRecord`: 任务记录，包含 goal、workspace_path、status 等字段
- `RunRecord`: 运行记录
- `EventRecord`: 事件记录
- `SessionRecord`: 会话记录，包含 task_id、agent_profile、workspace_mode、status 等
- `TurnRecord`: 轮次记录
- `ChallengeRecord`: 挑战记录
- `VerificationRecord`: 验证记录
- `LearningNote`: 学习笔记
- `SkillRecord`: 技能记录

枚举类型：
- `WorkspaceMode`: READONLY / SHARED / ISOLATED
- `TaskStatus`: pending / running / completed / failed
- `SessionStatus`: pending / running / completed / failed / needs_human
- `TurnPhase`: thinking / acting / done

#### `policy_gate.py` — 安全策略门控

`PolicyGate` 类实现命令和路径级别的安全检查：

```python
# 阻止的命令模式
BLOCKED_COMMANDS = [
    "rm -rf",           # 递归强制删除
    "git reset --hard", # 硬重置
    "git clean -fd",    # 强制清理
]

# 阻止的 Shell 内联执行
BLOCKED_SHELL_INLINE = [
    "python -c", "node -e", "sh -c"
]

# 保护的路径模式
PROTECTED_PATHS = [
    ".env", ".git/", "secrets/", "*.pem", "*.key"
]
```

#### `supervisor_loop.py` — 桥接自动监督者

- 轮询状态
- 运行验证
- 失败时发送挑战
- 成功时接受结果

### 5.5 Crew 编排层 (`crew/`)

这是系统的核心编排层。

#### `models.py` — Crew 领域模型

定义了丰富的 Crew 相关模型：

- **`CrewRecord`**: Crew 记录，包含 id、status、goal、worker_ids 等
- **`WorkerContract`**: Worker 契约，定义：
  - `capabilities`: 能力列表（如 edit_source、browser_e2e）
  - `authority_level`: 权限级别（readonly / write / admin）
  - `workspace_policy`: 工作空间策略（readonly / shared / isolated）
  - `write_scope`: 写入范围（路径列表）
  - `expected_outputs`: 期望输出
  - `acceptance_criteria`: 验收标准
- **`AgentProfile`**: Agent 配置文件，包含 `render_prompt()` 方法
- **`CrewEvent`**: Crew 事件
- **`WorkerTurnObservation`**: Worker 轮次观察
- **`DecisionAction`**: 决策动作
- **`AgentMessage`**: Agent 消息
- **`ProtocolRequest`**: 协议请求
- **`WorkerRecord`**: Worker 记录
- **`CrewTaskRecord`**: Crew 任务记录
- **`BlackboardEntry`**: 黑板条目

枚举类型：
- `CrewStatus`: created / running / completed / failed / stopped
- `WorkerRole`: explorer / implementer / reviewer / context_scout / source_write / patch_auditor / failure_analyst / guardrail_maintainer / browser_flow
- `WorkerStatus`: idle / busy / stopped / failed
- `AuthorityLevel`: readonly / write / admin
- `WorkspacePolicy`: readonly / shared / isolated
- `DecisionActionType`: spawn_worker / send_message / run_verification / issue_challenge / accept / stop_crew

| `status()` | 获取 Crew 状态 |
| `verify()` | 运行验证 |
| `challenge()` | 发出挑战 |
| `accept()` | 接受结果 |

| `changes()` | 获取变更列表 |

#### `decision_policy.py` — 动态决策引擎

`CrewDecisionPolicy.decide(snapshot)` 根据当前快照返回 `DecisionAction`。

决策逻辑基于优先级路由：
1. 检查是否有 Worker 需要生成
2. 检查是否有消息需要发送
3. 检查是否需要运行验证
4. 检查是否需要发出挑战
5. 检查是否可以接受结果

#### `gates.py` — 写入范围门控

`WriteScopeGate` 强制执行写入范围限制：

```python
# 受保护的模式
PROTECTED_PATTERNS = [
    ".git/", ".env", "secrets/", "*.pem", "*.key",
    "pyproject.toml",  # 锁文件
    "*.lock",
    ".github/workflows/"
]
```

返回 `GateResult`，状态为：
- `pass`: 通过
- `block`: 阻止
- `challenge`: 需要挑战确认

#### `scope.py` — 路径范围归一化

```python
    def list_by_turn(self, turn_id: str) -> list[AgentEvent]: ...
```

## 7. LLM Supervisor MCP Server（规划中）

> 详细设计文档: [2026-05-05-llm-supervisor-mcp-design.md](specs/2026-05-05-llm-supervisor-mcp-design.md)

### 7.1 概述

当前系统的 supervisor 是纯 Python 规则引擎 (`CrewDecisionPolicy`)，只能处理 6 种预设场景。规划将决策权交给 Codex (Claude Code)，通过 MCP Server 暴露 crew 管理能力为 tools。

### 7.2 战略/战术分层

| 层级 | 职责 | 执行者 |
|------|------|--------|
| 战略层 | 派谁、做什么、是否验收、是否 spawn 新 Worker | Codex (LLM) |
| 战术层 | 轮询 tmux、自动验证、自动挑战、超时检测、3 次升级 | Supervision Loop (Python) |

战术循环在战略决策点暂停返回给 Codex，规则引擎保留为 fallback（`auto_decide=True` 时自动触发）。

### 7.3 MCP Tools 清单

**Crew 生命周期**: `crew_start`, `crew_stop`, `crew_status`

**战略决策**: `crew_decide`, `crew_accept`, `crew_challenge`, `crew_spawn`

**上下文获取** (Context Layer，带压缩): `crew_blackboard`, `crew_events`, `crew_observe`, `crew_changes`, `crew_diff`

**战术执行**: `crew_run`, `crew_verify`, `crew_merge_plan`

### 7.4 Context Layer

核心价值——MCP Server 不是简单转发，而是对数据做智能压缩：

- `crew_status`: 返回结构化摘要（非原始 dump），每个 Worker 只保留 id/role/status/summary
- `crew_blackboard`: 按 worker_id/type 过滤，默认 limit=10，超过 20 条自动摘要
- `crew_events`: 只返回关键事件（turn.completed/failed/challenge/review），跳过中间事件
- `crew_observe`: 只返回当前轮次文本（非全量 tmux 快照）
- 每个 tool 有 `max_tokens` 参数（默认 2000），超过自动截断

### 7.5 黑板重新定位

| 维度 | 现在 | 改造后 |
|------|------|--------|
| 读取方 | CLI 命令（给人看） | Context Layer（给 Codex 看） |
| 读取方式 | 全量 dump | 按需过滤 + 自动摘要 |
| 上下文膨胀 | 有，无上限 | 无，Context Layer 控制 |

### 7.6 Supervision Loop 改造

当前 `CrewSupervisorLoop.run()` 是完整循环，改造为 `run_step()` 可暂停协程：

```python
@dataclass
class LoopStepResult:
    action: str          # "waiting" | "needs_decision" | "ready_for_accept"
    reason: str = ""     # 暂停原因
    context: dict        # 相关上下文
    snapshot: dict       # 供规则引擎 fallback 使用的快照
```

`crew_run` MCP tool 内部循环调用 `run_step()`，遇到战略决策点暂停返回给 Codex。

### 7.1 典型 Crew 运行流程

```

       │

       │

   ┌──────────────────────────────────┐
   │ a. 向 Worker 发送任务            │
   │ b. 等待 Worker 完成              │
   │    (轮询 tmux 窗格捕获标记)      │
   │ c. 解析 Worker 输出              │

   │ f. 评估结果                      │

   │       └─ 3次失败 → 学习笔记      │
   │ g. 检查是否所有任务完成          │
   │    └─ 未完成 → 回到 a            │
   └──────────────────────────────────┘
       │

       │
```

## 12. 已知缺点与不足

> **正在解决的问题**: 以下部分缺点正在通过 [LLM Supervisor MCP Server 设计](specs/2026-05-05-llm-supervisor-mcp-design.md) 解决：
> - 2.2 动态决策引擎的规则引擎局限性 → 升级为 LLM 决策
> - 2.6 黑板模式的上下文膨胀问题 → Context Layer 压缩
> - 12.1.1 两代架构并存 → MCP 层统一入口
> - 12.3.9 缺乏 Worker 优先级调度 → LLM 可智能调度

### 12.1 架构层面

1. **两代架构并存**: V3（JSONL 文件持久化）和 V4（事件溯源）同时存在，增加了代码复杂度和维护成本。部分模块（如 `crew/controller.py`）需要同时支持两种模式。

2. **缺乏运行时依赖管理**: `pyproject.toml` 声明无运行时依赖，但系统实际依赖 tmux、Claude CLI、Git 等外部工具，没有在包级别进行声明和检查。

3. **CLI 入口过于庞大**: `cli.py` 约 1150 行，包含所有命令的解析和执行逻辑，违反单一职责原则。应拆分为多个子命令模块。

4. **缺少统一的错误处理框架**: 各模块的错误处理方式不一致，有的抛出异常，有的返回错误码，有的记录日志后继续。缺乏统一的错误分类和恢复策略。

### 12.2 可靠性层面

5. **tmux 依赖的脆弱性**: 系统高度依赖 tmux 窗格捕获来检测 Worker 完成状态，这种方式容易受到终端输出格式变化的影响。标记检测是基于字符串匹配的，缺乏更强的结构化协议。

6. **崩溃恢复不完整**: 虽然 `recover_stale_busy_workers()` 可以恢复过期的 BUSY Worker，但对于 Worker 在关键操作中途崩溃（如写入文件、提交 Git）的情况，缺乏完善的回滚机制。

7. **乐观锁的局限性**: `transition_worker_status()` 使用 CAS 语义，但在高并发场景下可能出现活锁（多个 Worker 频繁重试同一状态转换）。

8. **事件存储 SQLite 的并发限制**: SQLite 事件存储在高并发写入场景下可能成为瓶颈，虽然提供了 PostgreSQL 替代方案，但切换配置不够便捷。

### 12.3 功能层面

9. **缺乏 Worker 优先级调度**: 当前的 Worker 选择是基于能力匹配和质量评分，缺乏优先级队列和抢占机制。所有 Worker 平等竞争，无法保证关键任务优先执行。

10. **学习系统的保守性**: 治理式学习要求人工审批才能激活新技能和护栏，这在快速迭代场景下可能成为瓶颈。缺乏自动化的渐进式激活策略。

11. **合并冲突处理简单**: `merge_arbiter.py` 仅检测文件所有权冲突，对于语义级别的合并冲突（如两个 Worker 修改同一文件的不同函数但存在逻辑依赖）无法处理。

12. **缺乏资源限制**: 没有对 Worker 的 CPU、内存、磁盘使用进行限制。长时间运行的 Worker 可能消耗过多系统资源。

### 12.4 可观测性层面

13. **日志体系不完善**: 缺乏结构化日志和日志级别控制。调试时需要手动检查 JSONL 文件和 tmux 会话。

14. **Web UI 功能有限**: 当前 UI 仅提供基本的状态查看功能，缺乏实时更新、交互操作和历史回放能力。

15. **缺乏性能指标收集**: 没有对 Worker 执行时间、验证通过率、挑战频率等关键指标进行系统性收集和分析。

### 12.5 安全层面

16. **策略门控可绕过**: `PolicyGate` 基于字符串匹配阻止危险命令，但可以通过编码、变量替换、管道等方式绕过。例如 `eval "rm -rf /"` 或 `cmd="rm -rf"; $cmd`。

17. **Worker 隔离不彻底**: 虽然使用 Git worktree 隔离代码，但所有 Worker 共享同一个文件系统和网络环境。恶意或错误的 Worker 可能影响其他 Worker 或宿主系统。

18. **缺少审计日志**: 安全相关的操作（如文件修改、命令执行、策略检查结果）没有专门的审计日志，难以进行事后追溯。

### 12.6 扩展性层面

19. **Protocol 接口版本管理缺失**: 虽然使用 Python Protocol 定义接口，但缺乏版本管理和向后兼容策略。接口变更可能导致现有适配器失效。

20. **Agent 包系统原始**: 当前的 Agent 包仅是 Markdown 文件片段，缺乏结构化的元数据、版本管理和依赖声明。

21. **缺乏插件机制**: 系统的扩展主要通过修改源码实现，缺乏标准化的插件接口让第三方扩展功能。

### 12.7 开发体验层面

22. **文档不完整**: 虽然有 `docs/superpowers/specs/` 和 `docs/superpowers/plans/` 目录存放设计文档，但缺乏面向用户的使用指南和 API 文档。

23. **本地开发环境搭建复杂**: 需要同时安装 tmux、Claude CLI、Git，并配置好环境变量，新开发者上手成本高。

24. **缺乏 CLI 自动补全**: argparse 构建的 CLI 没有 Shell 自动补全支持。
